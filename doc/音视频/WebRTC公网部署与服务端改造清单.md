# WebRTC 公网部署与服务端改造清单

## 1. 文档目的

本文基于当前项目已经采用的 WebRTC + WebSocket 信令模式，整理在“内网可用”基础上，将音视频能力发布到公网后需要补齐的服务端与部署侧工作。

重点回答两个问题：

1. 公网跨网络环境下，视频为什么内网能通但公网不一定能通
2. 从服务器侧开始，具体需要做哪些事情，是否需要安装额外的中继服务

---

## 2. 先给结论

### 2.1 你需要安装额外服务

是的。

如果你希望客户端在不同网络环境下也能稳定发起视频，除了你现有的：

- 业务服务端
- WebSocket 信令服务

之外，通常还需要额外部署：

- `STUN` 服务
- `TURN` 服务

实际生产里一般直接部署：

- `coturn`

`coturn` 同时可以提供：

- STUN
- TURN
- TURNS

也就是说，你说的“中转”应用，公网场景下通常就是：

- `TURN` 中继服务
- 最常用实现：`coturn`

---

### 2.2 你现有的 WebSocket 信令服务器不是媒体中继

你当前的 server WebSocket 只是做：

- 来电通知
- offer / answer 交换
- ICE candidate 交换

它不是音视频流转发服务器。

真正的音视频流是 WebRTC 自己通过 ICE 去建立的：

- 能直连就走点对点
- 不能直连就要走 TURN 中继

所以公网能否稳定通话，关键不在你现有信令逻辑本身，而在于：

- 是否有可用的 ICE 基础设施
- 尤其是否有 TURN

---

## 3. 为什么内网能打通，公网可能不行

内网中，设备之间通常：

- 网络可互访
- NAT 层数少
- candidate 中的 host 地址可直接使用

所以即使没有 TURN，也可能成功。

公网下常见场景：

- 家庭宽带 NAT
- 手机 4G / 5G NAT
- 企业网 / 校园网限制 UDP
- 双层 NAT
- 对称 NAT

这些环境下：

- 单靠本地地址不够
- 单靠 STUN 也不一定够
- 很多情况必须依赖 TURN 中继

因此，公网可用的标准答案基本都是：

- WebSocket 信令
- STUN
- TURN
- TLS

---

## 4. 服务端侧总任务清单

## 4.1 业务服务端

### 需要做的事情

- 保持你当前的 WebSocket 信令能力
- 确保公网域名可访问
- 支持 `wss://`
- 做 WebRTC 信令鉴权
- 做通话状态日志记录
- 做失败超时控制

### 重点

你现有业务服务端不一定要承担媒体转发。

它应该继续负责：

- 登录态
- 通话发起
- 通话接听
- offer / answer 交换
- candidate 转发
- 通话结束事件

---

## 4.2 TURN / STUN 服务

### 需要做的事情

- 单独部署 `coturn`
- 提供公网域名，如 `turn.xxx.com`
- 绑定公网 IP
- 配置用户名密码或动态凭证
- 支持 UDP / TCP / TLS

### 这部分才是真正的“中转”

如果两个客户端无法直接建立 P2P：

- 音视频数据就会通过 TURN 中继

也就是说：

- 业务 server 负责信令
- `coturn` 负责中继

---

## 5. 服务器侧实施顺序

## 5.1 第一步：先把业务服务端公网化

### 你需要做

- 给 server 配公网域名
- 配 `https`
- 配 `wss`
- 确保 WebSocket 能从公网正常连接

### 推荐形态

- `https://api.xxx.com`
- `wss://signal.xxx.com`

### 你要检查

- 反向代理是否支持 WebSocket Upgrade
- 证书是否正常
- 客户端是否不再使用内网 IP

---

## 5.2 第二步：安装 coturn

### 需要安装什么

建议安装：

- `coturn`

这是最常见、最成熟的 TURN 服务实现。

### 作用

- 提供 STUN
- 提供 TURN
- 提供 TURNS

### 结论

如果你问“是不是要安装应用来实现中转”，答案就是：

- 是，需要
- 最推荐装 `coturn`

---

## 5.3 第三步：开放服务器端口

至少需要考虑：

- `443/tcp`
- `3478/udp`
- `3478/tcp`
- `5349/tcp`
- 一段媒体 relay 端口范围，例如：
  - `49152-65535/udp`

### 你要配置的地方

- 云服务器安全组
- 系统防火墙
- 反向代理

### 注意

很多项目的问题不是 TURN 没装，而是：

- TURN 装了但安全组没放行

---

## 5.4 第四步：给客户端下发 ICE 配置

客户端必须在创建 `PeerConnection` 时配置：

- STUN
- TURN

示例：

```json
[
  { "urls": ["stun:turn.xxx.com:3478"] },
  {
    "urls": [
      "turn:turn.xxx.com:3478?transport=udp",
      "turn:turn.xxx.com:3478?transport=tcp",
      "turns:turn.xxx.com:5349?transport=tcp"
    ],
    "username": "turn_user",
    "credential": "turn_password"
  }
]
```

### 你服务端可以做的更优方式

- 不把 TURN 配置写死在客户端
- 通过接口动态下发

---

## 5.5 第五步：增加服务端日志与排查能力

建议你在 server 端记录这些内容：

- 谁发起了呼叫
- 谁接听了呼叫
- offer / answer 是否交换成功
- candidate 是否交换成功
- 通话时长
- 通话失败原因

### 推荐增加的日志字段

- `callId`
- `fromUserId`
- `toUserId`
- `messageType`
- `candidateCount`
- `reason`
- `timestamp`

### 目的

当用户说“能打电话但视频起不来”时，你至少能先判断：

- 是信令没到
- 还是 ICE 建连失败

---

## 6. 服务端专项 TODO List

## P0 必做

- [P0] 给业务 server 配置公网域名和 HTTPS
  - 目标：API 和 WebSocket 都能公网访问
  - 风险：客户端只能在内网环境使用

- [P0] 把 WebSocket 升级为 `wss`
  - 目标：公网、移动网络、企业网环境下更稳定
  - 风险：`ws` 在很多环境下容易被拦截或不稳定

- [P0] 部署 `coturn`
  - 目标：提供公网可用的 TURN / STUN
  - 风险：没有 TURN 时跨 NAT 场景大量失败

- [P0] 开放 TURN 所需端口
  - 目标：让 TURN 真正可用
  - 风险：服务已部署但实际不可连

- [P0] 在信令服务中确保完整转发：
  - `call/request`
  - `call/accept`
  - `call/end`
  - `offer`
  - `answer`
  - `candidate`
  - 风险：信令不完整会导致看起来“拨通了”，实际永远连不上

## P1 核心增强

- [P1] 增加信令鉴权
  - 目标：防止任意用户伪造呼叫或 candidate
  - 风险：公网暴露后安全风险上升

- [P1] 增加通话状态追踪
  - 目标：可观测、可排障
  - 风险：线上故障无法定位

- [P1] 增加超时控制
  - 目标：防止呼叫流程卡死
  - 风险：UI 长时间停留在连接中

- [P1] 提供 ICE 配置接口
  - 目标：服务端统一下发 TURN 配置
  - 风险：后续变更 TURN 节点需要客户端发版

## P2 生产优化

- [P2] TURN 动态鉴权
  - 目标：避免固定密码泄漏
  - 风险：TURN 被盗刷流量

- [P2] 多地域 TURN 节点
  - 目标：跨区域提升成功率和时延
  - 风险：单节点跨地域体验差

- [P2] 通话质量监控
  - 目标：形成线上质量画像
  - 风险：只能主观判断网络问题

---

## 7. 你现在最应该先做的事情

如果从“服务器这边开始”，建议你按下面顺序做：

1. 确认现有 WebSocket 信令服务已经可以公网访问
2. 把它切到 `wss`
3. 安装 `coturn`
4. 开放 TURN 端口
5. 给客户端加 `iceServers`
6. 做真实公网环境联调

也就是说：

- 第一阶段先让信令公网可用
- 第二阶段补 TURN 中继
- 第三阶段再做稳定性和监控

---

## 8. 最后的明确回答

### 你问：从服务器这边开始，我需要做什么？

答案是：

- 让你的信令服务公网可访问
- 支持 `wss`
- 部署 TURN 服务
- 开放 TURN 端口
- 提供给客户端正确的 ICE 配置

### 你问：是不是需要安装什么应用来实现中转？

答案是：

- 是
- 需要部署 TURN 中继服务
- 最常见、最推荐的实现就是：`coturn`

它不是替代你现有 server，而是作为 WebRTC 的网络穿透与中继基础设施，和你的业务 server 配合工作。

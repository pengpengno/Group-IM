# Redis Stream 分布式消息路由设计文档

## 1. 业务背景与痛点
在传统的单机 IM 系统中，消息投递非常简单：直接在内存中找到用户的 Socket 连接并发送即可。但在**多节点分布式场景**下，用户 A 连接在节点 1，用户 B 连接在节点 2。
*   **挑战**：节点 1 收到 A 的消息后，如何将消息准确、实时、且不丢失地传递给节点 2 上的用户 B？
*   **方案**：我们需要一个“中间人”来负责跨节点的消息中转，这就是**分布式消息路由**。

---

## 2. 核心技术：Redis Stream
我们选择了 Redis Stream 作为消息路由的底层设施。它不是简单的发布/订阅（Pub/Sub），而是一个**持久化的、支持消费者组**的消息队列。

### 2.1 关键概念（通俗理解）
*   **Stream (流)**：想象成一个只允许追加的“账本”，消息写进去就带了一个唯一 ID，除非主动清理，否则不会消失。
*   **Consumer Group (消费者组)**：一群人（节点）一起读账本，Redis 会记录谁读到了哪里，保证负载均衡。
*   **ACK (确认应答)**：节点读完消息并成功发给用户后，告诉 Redis“我搞定了”。
*   **Pending List (挂起列表)**：记录了“已取走但还没说搞定”的消息。如果节点突然挂了，这些消息就在 Pending 列表里，重启后可以重新处理。

---

## 3. 技术方案横向对比

| 特性 | Redis Pub/Sub | Redis List | **Redis Stream** | RabbitMQ / Kafka |
| :--- | :--- | :--- | :--- | :--- |
| **持久化** | ❌ 消息即发即失 | ✅ 支持 | ✅ **高** | ✅ 极高 |
| **消费者组** | ❌ 广播模式 | ❌ 只能单消费者 | ✅ **原生支持** | ✅ 原生支持 |
| **消息确认** | ❌ 无 | ❌ 取走即删除 | ✅ **ACK 机制** | ✅ 完善 |
| **开发难度** | 极易 | 容易 | **适中 (推荐)** | 复杂 (需引入新组件) |
| **应用场景** | 实时广播 | 简单异步任务 | **分布式路由 / 任务队列** | 海量数据处理 / 金融级 MQ |

**为什么选择 Redis Stream？**
IM 系统已经重度依赖 Redis（存储索引、状态、缓存），引入 Kafka 或 RabbitMQ 会增加运维成本。Redis Stream 提供了 MQ 的核心能力且无需额外中间件，是**轻量级分布式系统**的最佳选择。

## 4. 本工程落地实现逻辑

### 4.1 核心结构
每个服务器节点启动时，都会拥有一个专属的“信箱”（Stream）：
*   **Stream Key**: `im:route:<nodeId>` (每个节点只管自己的信箱)
*   **Consumer Group**: `im-group`
*   **Consumer Name**: `<nodeId>`

### 4.2 消费流程（手写长轮询模型）
为了保证极高的稳定性，我们弃用了 Spring 默认的容器，采用**手写 While 循环**控制：

1.  **初始阶段 (Handle Pending)**：
    *   启动时先用 `XREADGROUP ... STREAMS key 0` 读取“账本”中那些之前由于节点宕机或其他意外导致没处理完的消息。
    *   补发重试，直到 Pending 列表清空。
2.  **监听阶段 (Increment Loop)**：
    *   使用 `XREADGROUP ... STREAMS key > BLOCK 10s` 进入长轮询模式。
    *   如果有新消息，立即处理；如果没有，线程会挂起 10 秒等待，不会产生 Busy Loop（死循环空转）。
3.  **投递与确认**：
    *   解析消息 -> 查找本地内存连接 -> 发送。
    *   **只有成功发送给用户后**，才会调用 `XACK`。如果本地找不到连接或报错，消息会停留在 Pending 列表中。

---

## 5. 容错与高可用设计

*   **宕机不丢消息**：节点 A 发送给节点 B 的路由消息存放在 Redis 中。如果节点 B 正在重启，消息会堆积在 Stream 里，节点 B 上线后第一件事就是清空积压的消息。
*   **无感扩容**：当部署新节点时，新的 `im:route:<newNodeId>` 会被自动创建，路由算法（基于用户在线状态）会自动将消息导向新节点。
*   **异常重试逻辑**：在代码层面，所有网络抖动和 Redis 断连都有 `2秒` 的退避重试逻辑，防止因网络瞬断导致消费线程退出。

---

## 6. 其他建议（补充项）

*   **消息过期清理**：由于 Stream 是持久化的，建议定期执行 `XTRIM` 限制 Stream 长度（如 10000 条），防止 Redis 内存膨胀。
*   **本地缓存联动**：路由前先查本地连接。如果用户就在本节点，直接走本地投递（Loopback），绕过 Redis 以降低网络开销及延迟。
*   **死信处理**：如果某条消息连续 N 次读出来都投递失败（比如协议格式错误），应考虑手动移入 DLQ（死信队列），避免死循环。

# 多端入口与启动链路梳理

## 1. 文档目的

本文基于当前项目中的三个实际入口文件进行检索与整理：

- 服务端入口：`server/src/main/java/com/github/im/server/Application.java`
- Web/Desktop 端入口：`Group-app/Group-Electronjs/main/index.ts`
- 移动端入口：`Group-app/Group/composeApp/src/androidMain/kotlin/com/github/im/group/MainActivity.kt`

目标是说明当前多端项目的启动方式、初始化职责、关键依赖模块，以及三端之间的边界关系。

## 2. 总体结论

当前项目并不是三套完全独立的实现，而是按平台能力分层：

- 服务端以 Spring Boot 为主入口，同时在启动阶段额外挂起 TCP 长连接服务。
- Desktop 端以 Electron `main` 进程为入口，负责窗口、托盘、IPC 和桌面能力接入。
- Web 与 Desktop 共享大部分 renderer React 代码，但 Web 环境通过 `electronAPI` 的降级实现走纯浏览器能力。
- 移动端以 Android `MainActivity` 为入口，负责 Android 上下文、凭据存储、Koin 依赖注入、配置初始化和 Compose 根界面装配。

这说明当前工程已经形成比较明确的跨平台边界：

- Server 负责 HTTP API、认证、文件、消息与长连接服务。
- Electron Main 只负责桌面壳层能力，不直接承载业务 UI。
- Renderer/Compose 承载主要客户端业务界面。
- 移动端和桌面端都通过各自平台注入方式，把“存储、文件、音频、WebRTC、Socket”接到共享业务层。

## 3. 服务端入口梳理

### 3.1 入口文件职责

`Application.java` 是标准 Spring Boot 入口，核心职责有：

- 使用 `@SpringBootApplication(scanBasePackages = {"com.github.im.server", "com.github.im.common"})` 同时扫描 `server` 与 `common` 包。
- 开启异步能力：`@EnableAsync`
- 开启定时任务能力：`@EnableScheduling`
- 通过 `SpringApplication.run(Application.class)` 启动整个服务。
- 在 `ServerListener` 中监听 `WebServerInitializedEvent`，输出当前 HTTP 服务地址。

从这里可以看出，服务端并不只是一个单纯的 Controller 容器，而是把 `common` 里的通信能力也一起装配进来。

### 3.2 HTTP 服务与 TCP 长连接并行启动

虽然 `Application.java` 本身只显式启动了 Spring Boot，但实际还会继续拉起 TCP 服务：

1. `ConnectConfig.java` 中定义了 `ReactiveServer` Bean。
2. 该 Bean 实际来自 `ReactorTcpServer.getInstance()`。
3. 初始化时使用 `tcp.port` 配置，默认端口为 `8088`。
4. `ConnnectInitialize.java` 在拿到 Spring `ApplicationContext` 后调用 `reactiveServer.start()`。

因此，当前服务端启动后实际会同时具备两类能力：

- HTTP 服务：Spring Boot Web 容器，默认读取 `server.port`
- TCP 长连接服务：`ReactorTcpServer`，默认读取 `tcp.port`

这与 IM 场景是匹配的：HTTP 更适合登录、查询、文件元数据等请求；TCP 更适合消息、推送、连接态维护。

### 3.3 当前服务端提供的上层业务

从 `server/src/main/java/com/github/im/server/controller` 下的控制器分布看，当前服务端已经承载了这些主要业务：

- 认证与用户：`AuthController`、`UserController`
- 组织与公司：`CompanyController`、`CompanyUserController`、`OrganizationController`
- 会话与消息：`ConversationController`、`MessageController`
- 社交关系：`FriendshipController`、`GroupMemberController`
- 文件：`FileUploadController`、`FileMetaController`
- AI/MCP：`AiBotController`、`McpInfoController`、`McpDemoController`

说明服务端是整个项目的中心中台，既提供标准 REST API，也承担 IM 长连接后端。

## 4. Web / Desktop 端入口梳理

### 4.1 `main/index.ts` 的真实定位

你标注的 `Group-app/Group-Electronjs/main/index.ts` 更准确地说是 **Desktop 端主进程入口**，不是纯 Web 页面入口。

它的职责主要是：

- 创建 Electron `BrowserWindow`
- 注册托盘 `Tray`
- 保证单实例运行 `app.requestSingleInstanceLock()`
- 注册并初始化 IPC 能力
- 把 renderer 页面装入窗口

也就是说，它更像“桌面壳层启动器”，而不是 React 页面本身。

### 4.2 Desktop 启动链路

当前 Electron 主进程启动流程如下：

1. 入口文件加载时先引入 IPC Handler：
   - `auth-handler`
   - `file-handler`
   - `notification-handler`
   - `video-handler`
   - `socket-handler`
2. 立即执行 `registerVideoHandlers()`，注册桌面捕获相关 IPC。
3. `app.on('ready')` 时创建主窗口 `createWindow()`。
4. `createWindow()` 中配置：
   - `contextIsolation: true`
   - `nodeIntegration: false`
   - `preload: preload.js`
5. 窗口加载 `dist/index.html`
6. 窗口创建后执行 `initializeSocketHandler(mainWindow)`，把主进程 Socket 能力与窗口事件总线接通。
7. 同时创建系统托盘 `createTray()`。

这条链路体现出比较标准的 Electron 分层：

- `main/index.ts`：生命周期与系统资源管理
- `preload.ts`：安全桥接层
- `renderer/*`：实际 React 业务界面

### 4.3 `preload.ts` 的边界作用

`preload.ts` 通过 `contextBridge.exposeInMainWorld('electronAPI', ...)` 向 renderer 暴露安全 API，主要包括：

- 认证：`login`、`queryUsers`、`getUserCompanies`
- 文件：`uploadFile`、`downloadFile`、`selectFile`
- 通知：`showNotification`、`requestNotificationPermission`
- Socket：`socketConnect`、`socketSend`、`socketDisconnect`、`socketIsActive`、`socketSendMessage`、`socketMarkRead`
- 桌面共享：`getDesktopSources`
- Socket 事件订阅：`onSocketMessage`、`onSocketConnected`、`onSocketDisconnected` 等

这层设计符合 Electron 的跨端边界要求：

- React renderer 不直接访问 Node.js 或系统 API
- 所有系统能力都经由 IPC 暴露
- Web 环境可以在没有 `electronAPI` 的情况下走降级逻辑

### 4.4 Web 与 Desktop 的共享关系

项目里真正的 UI 挂载点在 `Group-app/Group-Electronjs/renderer/index.tsx`：

- 使用 `ReactDOM.createRoot(container)` 启动 React
- 根组件外包了一层 Redux `Provider`

这说明桌面端展示的页面，本质上就是 renderer React 应用。

同时 `renderer/services/api/electronAPI.ts` 明确表明该 renderer 兼容两种环境：

- 如果存在 `window.electronAPI`，则使用 Electron 注入的桌面能力
- 如果不存在，则回退到 `webAPI`

`webAPI` 当前提供了以下特征：

- 登录、用户查询走 HTTP API
- 通知走浏览器 `Notification`
- 文件选择走 HTML `<input type="file">`
- Socket 相关能力在纯 Web 环境下多数直接报错或返回不支持

再结合 `package.json` 中的脚本可见：

- `npm run dev` / `build` 用于 Electron 主进程 + renderer 构建
- `npm run web` / `build:web` 用于纯 Web 构建

因此当前 `Group-Electronjs` 目录本质上是 **一套共享 renderer + 两套运行壳（Electron / Web）** 的结构。

## 5. 移动端入口梳理

### 5.1 `MainActivity.kt` 的核心职责

Android 端入口 `MainActivity.kt` 负责在 Activity 生命周期开始时完成以下初始化：

1. 调用 `enableEdgeToEdge()`
2. 调用 `initAndroidContext(applicationContext)` 初始化 Android 平台上下文
3. 为全局凭据提供器设置实现：
   - `GlobalCredentialProvider.storage = AndroidCredentialStorage(applicationContext)`
4. 启动 Koin：
   - `androidContext(this@MainActivity)`
   - `modules(appmodule, commonModule)`
5. 从 Koin 中获取 `ConfigManager`
6. 异步执行 `configManager.initialize()`
7. 持续监听 `currentConfig`，并同步更新 `ProxyConfig`
8. 通过 `setContent { KoinContext { App() } }` 启动 Compose UI

因此，`MainActivity` 并不只是一个界面入口，而是 Android 平台适配层的总装入口。

### 5.2 Android 平台注入的能力

`appModule.kt` 中注册了大量 Android 侧实现，主要包括：

- 数据库：`AndroidDatabaseDriverFactory`
- 代理配置存储：`ProxyConfigStorage`
- 文件能力：`AndroidFilePicker`、`FileStorageManager`
- 音频能力：`AndroidAudioPlayer`、`AndroidVoiceRecorder`、`VoiceFileManager`
- 实时通信：`AndroidSocketClient`
- WebRTC：`AndroidWebRTCManager`
- 业务仓储：`UserRepository`、`ConversationRepository`、`ChatMessageRepository`、`FilesRepository` 等
- 业务管理器：`ChatSessionManager`、`LoginStateManager`
- 视图模型：`UserViewModel`、`ChatViewModel`、`ChatRoomViewModel`、`ContactsViewModel`、`VideoCallViewModel`、`VoiceViewModel`

这说明 Android 端采用的是：

- 平台层提供具体实现
- 业务层通过依赖注入使用抽象能力
- UI 层通过 ViewModel 消费业务能力

### 5.3 共享模块与 Compose 根界面

`commonModule` 当前至少提供：

- `ConfigManager`
- 当前配置快照
- `AudioPlaybackManager`

`App.kt` 是 Compose 根组件，当前主要负责：

- 初始化日志 `Napier.base(DebugAntilog())`
- 渲染视频播放器管理器 `VideoPlayerManager.Render()`
- 进入应用启动界面 `AppStartScreen()`

因此 Android 端的链路可以概括为：

`MainActivity` -> 平台初始化 -> Koin 注入 -> 配置初始化 -> Compose `App()` -> 启动页/主业务界面

## 6. 三端边界对照

### 6.1 服务端

- 负责统一业务规则、持久化、认证、文件、组织架构、消息接口
- 负责 HTTP API 与 TCP 长连接服务
- 是所有客户端共同依赖的后端中心

### 6.2 Desktop / Web

- Desktop 的 `main/index.ts` 只负责桌面能力接入与窗口生命周期
- 真实业务 UI 在 `renderer`
- Web 与 Desktop 共享 renderer 层
- Electron 独有能力通过 `preload.ts + ipcMain` 暴露

### 6.3 Android

- 由 `MainActivity` 负责注入 Android 平台能力
- 通过 Koin 把平台实现接入共享业务层
- 用 Compose 承载界面
- 与 Electron 不同，Android 的平台生命周期、存储和多媒体能力都由本地实现接管

## 7. 当前架构特点总结

当前工程已经具备比较清晰的多端组织方式：

1. 服务端是统一业务中心，且同时支持 HTTP 与 TCP 两条通信链路。
2. Electron 采用主进程、预加载脚本、renderer 三层结构，桌面边界比较明确。
3. Web 并不是独立代码库，而是复用 Electron renderer，并在 API 层做环境降级。
4. Android 通过 Koin 和平台实现把本地能力接到共享业务层，适合继续做 Kotlin Multiplatform 风格演进。

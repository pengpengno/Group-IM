# 多端音视频会议方案设计文档（可维护版本）

## 1. 目标与边界

### 1.1 目标

支持以下终端互通的视频通话：

| 终端 | 实现方式 |
|------|----------|
| Android | 原生（KMP + webrtc-kmp） |
| Web | 浏览器 WebRTC |
| Desktop | 使用系统浏览器打开 Web 会议页面 |

实现互通组合：
- ✅ Android ↔ Web
- ✅ Android ↔ Desktop  
- ✅ Web ↔ Desktop
- ✅ 全部互通

### 1.2 明确不做的事情

为保证可维护性：

❌ **不实现** Desktop 原生 WebRTC 客户端  
❌ **不实现** Electron 客户端  
❌ **不在 KMP 中封装 WebRTC 抽象层**

## 2. 总体架构

```
                   Android   
                  webrtc-kmp 
                 
                       
             signaling   (Protobuf over WS/TCP)
                       
          
   Web        Netty      Desktop    
 Browser          signaling        Browser    
 WebRTC API            WebRTC API 
```

### 核心原则

**Web 和 Desktop 完全使用同一套 Web 前端工程**

## 3. 客户端职责划分

### 3.1 Android（KMP 项目内）

**负责：**
- 音视频采集与渲染（webrtc-kmp）
- PeerConnection 生命周期
- ICE / SDP 处理
- 通过信令通道发送消息

**不负责：**
- 房间调度策略
- 成员管理规则

### 3.2 Web / Desktop（同一工程）

**负责：**
- 使用浏览器 WebRTC API
- 页面 UI
- 信令接入

**Desktop 特殊行为：**
- 仅通过默认浏览器访问会议页面

### 3.3 后端（Netty）

**负责：**
- 连接管理
- 房间成员管理
- 消息转发
- 鉴权

**不做：**
- 音视频转发（非 SFU）

## 4. KMP 在本方案中的正确定位

### KMP 只承担：
- 业务模型
- 消息模型
- 信令流程控制
- 状态机

### 明确：
**KMP 不封装 WebRTC API**

## 5. 会议访问模型（统一入口）

### 5.1 创建会议

**客户端调用：**
```http
POST /api/meeting/create
```

**返回：**
```json
{
  "roomId": "12345",
  "meetingToken": "abc",
  "meetingUrl": "https://meeting.xxx.com/room/12345?token=abc"
}
```

### 5.2 Desktop 行为

桌面端点击会议：
1. 打开系统默认浏览器
2. 访问 meetingUrl

### 5.3 Web 页面启动流程

1. 从 URL 读取 token
2. 请求后端换取会话身份
3. 建立信令连接

## 6. 信令协议统一（核心可维护点）

### 6.1 协议统一使用 Protobuf

**单一 proto：**
```
meeting_signaling.proto
```

**所有端统一生成：**
- Android：Kotlin
- Web：protobufjs

### 6.2 推荐的最小消息集

```protobuf
message JoinRoomRequest {
  string roomId = 1;
}

message JoinRoomResponse {
  repeated Participant participants = 1;
}

message OfferMessage {
  string from = 1;
  string to   = 2;
  string sdp  = 3;
}

message AnswerMessage {
  string from = 1;
  string to   = 2;
  string sdp  = 3;
}

message IceCandidateMessage {
  string from = 1;
  string to   = 2;
  string candidate = 3;
  string sdpMid = 4;
  int32 sdpMLineIndex = 5;
}
```

### 6.3 信令通道

**Web 端使用：**
- WebSocket

**Android 端可继续使用：**
- Reactor TCP / Netty client
- （只要消息结构一致）

## 7. 呼叫流程（统一逻辑）

### 7.1 入会流程

```
JOIN
 ↓
服务端返回当前成员列表
 ↓
为每个已有成员建立 PeerConnection
```

### 7.2 连接建立规则（强制）

**为避免冲突：由 join 较晚的一方创建 offer**

**流程：**
1. A 已在房间
2. B 加入
3. B 创建 offer → A
4. A 返回 answer
5. 双方开始 ICE 交换

## 8. 通话状态机（强烈建议统一）

在 KMP commonMain 中定义状态机：

```kotlin
enum class MeetingState {
    IDLE,           // 空闲
    JOINING,        // 加入中
    NEGOTIATING,    // 协商中
    CONNECTED,      // 已连接
    DISCONNECTED,   // 已断开
    ENDED           // 已结束
}
```

**状态迁移仅由：**
- 信令事件
- WebRTC 回调事件
- 触发

## 9. Android 实现规范（webrtc-kmp）

### 9.1 创建 PeerConnection

```kotlin
val factory = PeerConnectionFactory.create()

val pc = factory.createPeerConnection(
    RtcConfiguration(listOf(
        IceServer(listOf("stun:stun.l.google.com:19302"))
    )),
    observer
)
```

### 9.2 生成 offer

```kotlin
pc.createOffer { sdp ->
    pc.setLocalDescription(sdp)
    signaling.sendOffer(targetId, sdp.description)
}
```

### 9.3 处理 answer

```kotlin
pc.setRemoteDescription(
    SessionDescription(Type.ANSWER, sdp)
)
```

### 9.4 ICE

```kotlin
override fun onIceCandidate(candidate: IceCandidate) {
    signaling.sendCandidate(
        IceCandidateMessage(
            from = localUserId,
            to = targetUserId,
            candidate = candidate.sdp,
            sdpMid = candidate.sdpMid,
            sdpMLineIndex = candidate.sdpMLineIndex
        )
    )
}
```

## 10. Web 实现规范（浏览器 WebRTC）

### 10.1 创建 PeerConnection

```javascript
const pc = new RTCPeerConnection({
  iceServers: [
    { urls: "stun:stun.l.google.com:19302" }
  ]
});
```

### 10.2 创建 offer

```javascript
const offer = await pc.createOffer();
await pc.setLocalDescription(offer);

sendOffer({
  type: 'OfferMessage',
  from: localUserId,
  to: targetUserId,
  sdp: offer.sdp
});
```

### 10.3 设置 answer

```javascript
await pc.setRemoteDescription({
  type: "answer",
  sdp
});
```

### 10.4 ICE

```javascript
pc.onicecandidate = e => {
  if (e.candidate) {
    sendCandidate({
      type: 'IceCandidateMessage',
      from: localUserId,
      to: targetUserId,
      candidate: e.candidate.candidate,
      sdpMid: e.candidate.sdpMid,
      sdpMLineIndex: e.candidate.sdpMLineIndex
    });
  }
};
```

## 12. 可维护性关键设计点

### 12.1 不共享 WebRTC 实现

**禁止尝试：**
- 在 KMP 中抽象 WebRTC API
- 在 Web 端模拟 Android API

### 12.2 只共享三样东西

1. **proto** - 统一的消息协议
2. **呼叫流程规则** - 统一的业务逻辑
3. **状态机设计** - 统一的状态管理

### 12.3 Peer 管理统一规则

**建议统一为：**
```kotlin
// KMP commonMain
class PeerManager {
    private val peers = mutableMapOf<String, PeerConnection>()
    
    fun getOrCreatePeer(remoteUserId: String): PeerConnection {
        return peers.getOrPut(remoteUserId) {
            createPeerConnection(remoteUserId)
        }
    }
    
    fun removePeer(remoteUserId: String) {
        peers.remove(remoteUserId)?.close()
    }
}
```

## 14. 为什么此方案可维护

| 维度 | 说明 |
|------|------|
| 技术复杂度 | 最低可行 |
| 客户端数量 | 2 个（Android + Web） |
| Desktop | 零代码 |
| 协议 | 单一 proto |
| 升级空间 | 可平滑引入 SFU |
| 人员扩展 | Web 与移动可独立开发 |

## 15. 强制工程规范（给 agent）

🚫 **禁止在 commonMain 中引用 WebRTC API**  
🚫 **禁止在 Android 端出现 Web 端逻辑**  
✅ **所有信令消息必须来自 proto**  
✅ **所有房间与流程判断只能在信令层**  
✅ **Peer 只负责连接，不负责业务决策**

## 17. 总结

本方案刻意放弃：
- 桌面原生客户端
- 全端统一 WebRTC 抽象

换取：
- 明确边界
- 长期可维护
- 低技术债

这是最符合你当前系统规模与技术栈的工程化选择。
